## 問題：並列の世界に線形思考で挑む

Agent ワークフローの多くは、厳密に順番通りに実行する必要はありません。典型的な「調査して公開する」パイプラインを例にすると、ある Agent が市場データを収集している間に、別の Agent が記事のアウトラインを作成できます。最終的な統合ステップだけが、両方の出力を必要とします。

すべてを線形に dispatch するなら — ステップ 1、待機、ステップ 2、待機、ステップ 3 — パフォーマンスを無駄にしています。しかし、依存関係を無視して並列化すると、上流の出力が必要なステップが失敗するか、不完全な結果を生み出してしまいます。

鍵は、**どのステップが独立しているか**、そして**上流タスクからの共有コンテキストが必要なのはどのステップか**を明確にすることです。

## dispatch の前にワークフローを図示する

依存関係のグラフを先にスケッチしてください。同じ層のステップは並列実行できます。下の層のステップは上の層を待つ必要があります。

```
intel-gather（hisui）    outline-draft（kohaku）
        \                        /
         \                      /
          → final-article（kohaku） → post-to-discord（spinel）
```

最上層の 2 つのブランチは並列実行。最終記事のステップは両方が完了してから開始。投稿ステップは記事の完成を待ちます。

## 並列ステップを一度に起動する

```bash
# 2 つの独立したステップを同時に起動
INTEL_ID=$(tetora dispatch --task "市場情報を収集" --agent hisui --json | jq -r '.id')
OUTLINE_ID=$(tetora dispatch --task "記事アウトラインを作成" --agent kohaku --json | jq -r '.id')

# 最終ステップは両方の上流タスクが完了するまで待機
tetora dispatch --task "記事を執筆" --agent kohaku \
  --depends-on "$INTEL_ID,$OUTLINE_ID" \
  --on-failure abort
```

Tetora は `hisui` と `kohaku` を同時に起動します。記事執筆ステップは **両方が完了する**まで `waiting` 状態のままです。それぞれ 5 分かかる 2 つのステップが、合計 5 分で完了します — 10 分ではありません。

## ステップ間でコンテキストを受け渡す

完了した各タスクは、共有のタスクストアに出力を書き込みます。`inherit_outputs` で上流タスクの ID を参照すると、次の Agent を起動する前に自動的に上流の結果がプロンプトに注入されます：

```json
{
  "task": "記事を執筆",
  "agent": "kohaku",
  "dependsOn": ["intel-task-id", "outline-task-id"],
  "context": {
    "inherit_outputs": ["intel-task-id", "outline-task-id"]
  }
}
```

手動でコピー＆ペーストする必要はありません。Tetora が上流の出力を次のステップに自動的に渡します。

## 活用のヒント

- 並列ブランチは互いに独立させてください — 2 つのステップが同じファイルに書き込むと競合が発生します
- 複雑なパイプラインを実行する前に `tetora workflow visualize` で DAG をレンダリングして確認しましょう
- タスク名は具体的に — `tetora task status` に表示され、デバッグが容易になります
- 通知ステップには `--on-failure continue` を組み合わせて、上流が失敗してもチームに必ず通知が届くようにしましょう

よく設計されたマルチステップワークフローは、きれいなコードと同じです：一目で構造が分かり、各部分が一つの明確な役割を持っています。
