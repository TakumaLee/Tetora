## 問題：毎回すべてを忘れるエージェント

デフォルトでは、エージェントのセッションはコールドスタートします。昨日何を話したか、先週どんな決定を下したか、現在のプロジェクトの状態はどうかを知りません。変わっていない文脈を毎回説明するために、セッションの冒頭を費やすことになります。

これは単なる不便ではなく、複利的なコストです。プロジェクトが長くなるほど、エージェントに必要なキャッチアップが増えていきます。

---

## Tetora の2層メモリシステム

Tetora はメモリをアクセスパターンの異なる2つの層に分けています：

| 層 | 場所 | 読み込み | 最適な用途 |
|---|---|---|---|
| **Auto-memory** | `~/.claude/.../memory/` | セッションごとに自動 | セッションをまたぐ事実・好み・学習済みルール |
| **Workspace memory** | `memory/`（プロジェクトルート） | エージェントが明示的に `Read` | 調査資料・エージェント日誌・ドメインデータ |

この分離には意味があります。Auto-memory は小さく常時存在し、エージェントの思考方法を形作ります。Workspace memory は大きく意図的なもので、必要なときだけ引き出します。

---

## Auto-Memory の設定

Auto-memory ファイルはプレーンな Markdown で、セッション開始時に読み込まれます。すべてのエージェントに知ってほしい事実を追加しておきましょう：

```markdown
<!-- memory/auto/project-context.md -->
# プロジェクト：Tetora
ステータス：開発中、ベータユーザーオンボード済み
技術スタック：Go バックエンド、Astro フロントエンド、Claude API
決定ログ：ログ解析には Haiku を使用（2026-04-11）
Owner の好み：システム出力に絵文字を使わない
```

グローバルな事実は `~/.claude/memory/`、プロジェクト固有の事実は `<project>/.claude/memory/` に配置します。

---

## セッション中にメモリへ書き込む

エージェントが保存すべき何か——ユーザーの修正、新しい制約、重要な決定——を発見したとき、適切なメモリファイルに書き込む必要があります：

```bash
# プロジェクトメモリに新しい学びを追記
echo "- Polymarket API レート制限：10 req/s（2026-04-25 確認）" \
  >> .claude/memory/domain-facts.md
```

実際には、エージェントは修正や記録すべきパターンを検知すると自動的にこれを行います。手動でトリガーすることもできます：

```
/remember staging DB は port 5433 を使用（5432 ではない）
```

## Workspace Memory：必要なときに引き出す

より大きな構造化された文脈——過去の調査、エージェント日誌、週次レポート——には Workspace memory を使います。エージェントは明示的にロードします：

```markdown
<!-- エージェントの Soul ファイル内 -->
## セッション開始時
- Polymarket 関連のタスクなら memory/domain/polymarket-notes.md を読む
- 新しい調査タスクを計画するなら memory/agents/hisui/diary.md を読む
```

これにより、セッションの文脈をデフォルトで軽量に保ちつつ、必要なときだけ深くできます。

---

## 得られる成果

メモリがしっかり管理されたエージェントは、ステートレスなツールから長期的なコラボレーターへと変わります。文脈が積み上がり、各セッションは前のセッションの上に構築され、決定は引き継がれ、好みは定着します。

**原則：同じことをエージェントに2回言いそうなら、それはメモリファイルに入れるべき情報です。**
